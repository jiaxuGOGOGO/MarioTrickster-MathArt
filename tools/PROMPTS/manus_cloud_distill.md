# Manus Cloud Distillation Protocol for MarioTrickster-MathArt

本文件是专门提供给云端智能体使用的**指令资产**。当用户在对话中上传一本书、论文、讲义、设计文档或其它数学知识素材，并明确要求将其中可落地的数学规则蒸馏进本仓库时，云端执行者必须把本文件当作首要执行规范。目标并不是生成一段泛泛而谈的摘要，而是把外部资料中可验证、可结构化、可持续复用的数学约束，转换成 MarioTrickster-MathArt 可吸收的知识资产，并安全、完整、可审计地提交到当前仓库。

## 1. Mission Contract

云端执行者需要先完整理解上传资料的主题、适用边界、公式条件、变量定义、适用对象和失败条件，然后再决定是否落地到仓库。任何时候都**不得**跳过研究与验证步骤直接编写知识文件，更不得把含糊不清、未经校验的“灵感描述”直接写入 `knowledge/`。如果资料本身含有明显错误、缺页、定义不完整或前后矛盾，必须先在产出中标记不确定性，再决定是否暂缓落地。

## 2. Required Output Targets

云端执行者在蒸馏完成后，只能将知识同步到以下知识载体或其直接子路径中：

| 路径 | 用途 | 约束 |
|---|---|---|
| `knowledge/` | 数学规则、研究笔记、蒸馏结果 | 必须是结构化、可审阅的文本资产 |
| `PROJECT_BRAIN.json` | 项目任务状态与会话状态 | 必须保持 JSON 语法合法 |
| `SESSION_HANDOFF.md` | 面向下一个操作者的上下文交接 | 必须明确本轮新增能力、边界、风险与下一步 |
| `tools/PROMPTS/` | 指令资产与提示词协议 | 必须可复用、可版本管理、可审计 |

云端执行者**绝对禁止**使用 `git add .`、`git add -A` 或任何会把缓存、模型权重、下载产物、临时输出和无关文件一起打包的宽泛命令。提交范围必须保持最小化、白名单化与可解释。

## 3. Distillation Workflow

整个工作流必须严格按下表执行，不能跳步。

| 阶段 | 目标 | 必做动作 |
|---|---|---|
| A. 研究 | 吃透资料原意 | 通读上传资料，提取核心公式、变量、前提假设、反例与适用边界 |
| B. 建模 | 转为项目知识 | 将规则映射到本项目已有知识结构、命名风格与约束格式 |
| C. 校验 | 阻断错误知识入库 | 检查 JSON/Markdown 语法、命名一致性、引用完整性与路径合法性 |
| D. 同步 | 落地到仓库 | 仅对白名单路径执行 Git 暂存、语义化提交与推送 |
| E. 交接 | 保证下一轮可持续推进 | 更新 `SESSION_HANDOFF.md` 与 `PROJECT_BRAIN.json` 的状态与待办 |

## 4. Rule Extraction Standard

当资料中出现数学定理、经验公式、参数约束或算法规则时，云端执行者必须把它们整理为**项目可执行的约束语义**，而不是照抄原文。每条规则至少需要包含以下信息：规则名称、适用域、输入变量、输出变量、约束表达式、适用前提、风险提示、与现有知识的关系。如果原文有图表、单位或限定条件，也必须一起转译进规则说明，而不是只留下公式本身。

如果上传的是书籍或长文档，执行者必须先进行章节级筛选，只保留能直接指导本项目实现的内容。对于与项目目标无关的背景史、闲谈、重复案例和营销措辞，应当主动舍弃，避免污染知识库。

## 5. Validation Gates

在任何提交动作发生前，执行者必须完成如下校验。

| 校验项 | 说明 |
|---|---|
| 语法校验 | 所有 `.json` 文件必须能够被本地解析，所有 Markdown 文件必须非空且结构清晰 |
| 路径校验 | 待提交文件必须全部位于白名单目录中 |
| 语义校验 | 规则不得与本项目既有命名、数据结构或会话状态明显冲突 |
| 安全校验 | 绝不能把 API Key、Token、个人隐私或外部系统凭证写进仓库 |
| 范围校验 | 提交只包含本次研究真正相关的知识资产，不得夹带缓存或临时文件 |

只要有任何一项校验失败，执行者就必须停止提交，并把失败原因明确写入交付说明或交接文档。

## 6. Commit and Push Policy

提交信息必须是语义化、可追溯、带会话上下文的文本。推荐格式如下：

> `SESSION-XXX: distill <topic> into knowledge assets`

如果提交失败、推送冲突或权限不足，执行者应当保持主流程稳定，报告需要人工介入的原因，但不得让整个知识蒸馏任务以“程序崩溃”的形式结束。冲突处理时必须优先保留用户最新明确指令，其次再考虑与远端历史的合并策略。

## 7. Cloud-to-Repo Handoff Rules

当云端执行者已经拥有仓库读写能力时，应在推送前完成以下最小对照：

1. `knowledge/` 中新增或修改的文件是否真的对应本次上传资料。
2. `PROJECT_BRAIN.json` 的任务状态、`last_session_id`、`last_updated` 与待办列表是否同步更新。
3. `SESSION_HANDOFF.md` 是否清晰写明：本轮研究主题、已落地内容、人工干预边界、剩余风险、下一步建议。
4. 是否遗漏了与本次落地强相关的辅助文件，例如研究纪要、协议文档或测试文件。

只有在上述对照全部通过后，才允许执行最终同步。

## 8. Failure Handling

当资料歧义过大、仓库状态异常、远端冲突严重或本地验证失败时，执行者必须输出一份清晰的阻塞说明。说明中应明确指出：哪一步失败、失败证据是什么、需要用户补充什么信息、哪些文件已经完成、哪些文件尚未安全提交。禁止在不确定状态下“赌一把直接推送”。

## 9. Definition of Done

只有同时满足以下条件，任务才算真正完成：研究完成、规则完成、校验完成、白名单提交完成、远端推送完成、`PROJECT_BRAIN.json` 更新完成、`SESSION_HANDOFF.md` 更新完成。只完成其中一部分，不得宣称“已经全部同步”。
