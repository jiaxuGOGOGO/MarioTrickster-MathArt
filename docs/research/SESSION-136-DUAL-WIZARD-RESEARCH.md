# SESSION-136 Research: Dual Wizard, Local/Cloud Distillation Bus, and GitOps Knowledge Sync

本轮改造先对外部工程与研究资料进行了对齐，以确保后续代码落地不是拍脑袋式拼接，而是建立在可解释、可审计、可持续演进的共识之上。研究结论表明，本项目新增的“本地科研蒸馏”与“云端 Manus 直推蒸馏”不应被实现为两套彼此割裂的脚本，而应共享同一套**强类型会话上下文、模式权限边界、知识白名单提交代理与 Prompt-as-Code 指令资产**。这样做既能保留本地侧对 API 密钥、离线审计和资源控制的优势，也能释放云端侧在长链推理、文件解析和 GitHub 直推方面的能力。[1] [2] [3] [4] [5]

从配置治理角度看，最直接的约束来自 Twelve-Factor 对配置的定义。凡是会随部署变化的外部服务凭证、服务地址、模型名称与运行参数，都必须与代码严格分离；一个健康系统应当满足“代码库可随时开源而不泄露任何凭证”的检验标准。[1] 这意味着本轮新增的本地科研蒸馏配置不能硬编码进 Python 模块，也不能依赖散落且容易误提交的临时文件。更稳妥的实现方式，是提供一个终端配置向导，以 `.env` 或 `config.local.json` 这类**本地非追踪配置载体**来保存 API Key、Base URL 与 Model Name，并在写入前后都强制检查 `.gitignore`，把“防泄露”从文档提醒升级为代码级断言。[1]

从启动性能与模式隔离角度看，显式惰性导入是本次调度总线的关键设计原则。PEP 810 强调，惰性导入应当是**显式、局部、受控且可渐进采用**的工程决策，尤其适用于命令行工具、多子命令工具链以及拥有大型依赖图的系统。[2] 对本项目而言，这一原则直接导向一个明确结论：顶层入口不能全局导入 `torch`、ComfyUI 客户端或其它重型 AI 依赖，而应当通过 `mode_dispatcher.py` 在模式真正被选中后才进行按需加载。这样不仅能保证 `--help`、纯 CPU 审计和本地科研向导的秒级冷启动，也能从架构上避免“为了看菜单就先把 GPU 守护进程拉起来”的错误行为。[2]

从模式治理和内聚性角度看，双轨总线最适合采用注册表友好的策略分发模型，而不是在中心入口继续堆砌新的 `if/elif`。本项目已有明显的 registry-oriented 基础设施，因此新增生产、进化、本地科研蒸馏与 CPU dry-run 四类会话时，应当进一步把**模式选择**与**模式实现**拆开：前者由 CLI 向导或无交互参数解析完成，后者由强类型 `SessionContext` 描述是否需要 GPU、是否允许启动 daemon、是否只读知识库、是否允许 Git Push 等权限边界。这样可以把“防模式交叉污染”内化为运行时契约，而不只是人工注意事项。[2]

从知识同步与远端可追溯性角度看，OpenGitOps 的四项原则为本项目提供了非常直接的参考框架。一个符合 GitOps 精神的系统，必须让目标状态以声明式形式被表达、以版本化且不可变的方式保存在 Git 中、由软件代理自动拉取或提交，并在持续对账中收敛到期望状态。[3] 将这一原则翻译到 MarioTrickster-MathArt，结论就是：知识蒸馏得到的规则文件、脑图状态文件和会话状态文档不应停留在本地临时目录，而应当作为**知识即代码**资产进入仓库，由 `git_agent.py` 以严格白名单方式提交。与此同时，代理必须避免 `git add .` 这类灾难性操作，只允许 `knowledge/`、`PROJECT_BRAIN.json`、`SESSION_HANDOFF.md` 与必要的提示协议资产进入打包范围，从而满足“可审计”与“防大文件推云”两条底线。[3]

从云端蒸馏协议角度看，版本化提示词资产同样需要遵守工程化原则。GitHub 的提示工程实践文件明确强调，提示词应像代码一样被版本管理，记录变更理由，并在可能时维持兼容性。[4] 因而，`tools/PROMPTS/manus_cloud_distill.md` 不能只是一个随手写的说明文档，而应当是一份**可复用、可审查、可迭代**的指令资产：它需要规定输入来源、知识抽取步骤、目标约束格式、语法校验要求、允许提交的路径、提交消息格式，以及发生冲突或校验失败时的回退策略。只有这样，云端 Manus 直推才不会成为一条高风险“黑箱捷径”，而会变成与本地蒸馏同样可治理的正式通道。[4]

从总体架构选择上看，边云协同生成式 AI 研究也支持本项目采用双轨而非单轨设计。相关研究指出，大模型云端能力与小模型或本地能力并不是互斥关系，而应按任务特性进行协同分配；本地侧适合隐私敏感、低延迟或设备相关控制，云端侧更适合重推理、全局整理与知识沉淀，中间结果则需要以结构化资产回传与复用。[5] 映射到当前项目后，可以把本地科研蒸馏视为“用户自持密钥、在本机执行实验与校验”的轨道，把云端 Manus 直推视为“用户在对话中上传资料，由云端智能体直接解析、验证并提交知识资产”的轨道。二者共用统一知识格式与 GitOps 审计面，才能真正形成三层进化闭环，而不是两条互不相认的流水线。[5]

综合以上研究，本轮落地应遵守如下约束。配置必须本地化且默认不追踪；模式调度必须显式惰性导入；生产模式必须只读知识库；科研与审计模式不得无故拉起 GPU 守护进程；知识同步必须使用白名单与本地语法验证；云端协议必须沉淀为仓库内指令资产。只要这些约束被写入代码与测试，而不是停留在文字层面，项目就能在保证安全边界的前提下，获得真正可演进的双轨蒸馏能力。
