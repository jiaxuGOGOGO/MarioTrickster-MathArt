## Python Packaging User Guide — Creating and discovering plugins

- Python 官方建议的插件发现机制有三类：**命名约定**、**命名空间包**、**包元数据/entry points**。
- 对当前仓库而言，现有 **装饰器自注册** 已经能完成仓库内插件接入；如果未来要支持仓库外插件扩展，最稳妥的增强方向是新增一个**可选 entry-point 发现层**，而不是替换现有装饰器注册表。
- 官方文档特别提醒：不要让主顶层包直接作为插件命名空间，否则单个坏插件可能导致整个包不可导入。这个结论支持本项目继续把外部发现层做成**隔离可选层**，而不是把核心 `mathart` 顶层改造成脆弱的全局命名空间插件入口。
- 可执行结论：本轮 Golden Path 更应做“**现有注册表硬化 + 可选外部发现兼容**”，不应推翻 SESSION-064 已经建立的装饰器注册模式。

## GitHub Docs — `schedule` 事件

| 结论 | 对 Golden Path 的直接含义 |
|---|---|
| 定时工作流运行在默认分支的最新提交上 | 每日注册表 E2E 守护应放入默认分支上的 workflow 文件，不能只留在本地脚本 |
| 工作流在整点高峰可能延迟甚至掉队 | 不应把每日扫描放在整点；应选择偏移时间以降低调度抖动 |
| 公开仓库 60 天无活动会自动禁用定时任务 | 守护流程文档中应记录这一限制，并保留 `workflow_dispatch` 作为人工补跑入口 |
| 最短调度周期为 5 分钟，使用 POSIX cron | 本项目每日巡检完全可以用单一 cron 表达，并额外保留手动触发 |

结论上，本轮应实现“**本地可运行的注册表全量 E2E 脚本 + GitHub Actions 每日调度 + 手动补跑入口**”的双层守护，而不是只写一个无人调用的测试脚本。
