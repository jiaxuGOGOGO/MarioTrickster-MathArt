# MarioTrickster-MathArt 工作协议与进化准则

本文档定义了人类开发者（你）与 AI 代理（我）在本项目中的协作模式、职责边界以及系统自进化的核心准则。本协议的目的是确保项目在**尽量自动化**的前提下，实现**高质量的不断进化迭代**，同时避免无效劳动和知识冗余。

---

## 1. 协作模式与职责划分

我们的协作基于"异步反馈与闭环迭代"模式。

### 1.1 AI 代理的职责（我的工作）
- **代码实现与架构设计**：根据你的需求，编写、重构和优化项目的 Python 核心代码。
- **知识蒸馏与去重**：解析你上传的 PDF/Markdown 资料，提取数学规则和美术约束，并通过 `DeduplicationEngine` 确保知识库不冗余、不遗漏。
- **数学模型挖掘**：通过 `MathPaperMiner` 主动在全网（arXiv、GitHub 等）挖掘与像素美术、程序化生成相关的最新数学模型论文，并评估其接入价值。
- **自循环迭代与质量评估**：在 CPU 环境下运行内循环，利用数学评估器（清晰度、色彩和谐度、风格一致性等）自动寻找最优参数。
- **无效迭代拦截**：通过 `StagnationGuard` 监控迭代过程。如果发现生成的图像在视觉上没有实质性改变（pHash 距离过小），将立即停止无效迭代，尝试自动修复或生成诊断报告。

### 1.2 人类开发者的职责（你的工作）
- **提供算力与外部环境**：在需要 GPU 算力（如可微渲染、Unity Shader 编译）时，在本地运行我推送的代码，并将结果（如截图、日志）反馈给我。
- **上传高质量知识源**：持续提供包含美术理论、数学模型、游戏设计经验的 PDF 书籍或论文，供我进行知识蒸馏。
- **仲裁与决策**：当 `StagnationGuard` 报告"数学约束与美术目标互斥"或"参数空间耗尽"且 AI 无法自动修复时，根据我提供的诊断报告，给出明确的优化方向或调用外部 API 协助判断。
- **明确进化方向**：随时提出新的宏观需求（如"开始支持伪 3D 表现"、"接入 Unity Shader 优化"）。

## 2. 核心进化机制

### 2.1 拒绝无效迭代（StagnationGuard）
**原则**：绝不为了迭代而迭代。如果参数改变但视觉效果不变，或者质量分数停滞，必须立即干预。

- **检测标准**：连续 N 次迭代中，质量分数提升 < 0.005，且图像感知哈希（pHash）差异 < 12.5%。
- **处理流程**：
  1. **自动诊断**：系统分析原因是"参数空间过窄"、"生成器忽略参数"还是"数学与美术互斥"。
  2. **AI 仲裁**：调用内部 LLM 接口，尝试给出修改建议（如放宽某项约束）。
  3. **人类介入**：如果 AI 无法解决，系统将生成 `STAGNATION_LOG.md` 报告并暂停，等待你通过对话框给出明确指令。

### 2.2 知识蒸馏与去重（DeduplicationEngine）
**原则**：避免重复知识，但绝不因为害怕重复而漏掉有价值的细微差别。

- **三级去重机制**：
  1. **精确匹配**：哈希值完全一致的规则，静默跳过。
  2. **语义相似**：余弦相似度 > 0.85 的规则，**保留两者**并标记为变体（Variant），以防丢失细微的上下文差异。
  3. **参数重叠**：如果新规则与旧规则修改同一个数值参数且差异 < 5%，则自动取交集/平均值进行合并。
- **连续性保障**：每次蒸馏会话都有唯一编号（如 `DISTILL-004`）。即使你在全新的对话框中上传书籍，系统也会读取历史记录，无缝衔接，不会断档。

### 2.3 数学模型主动挖掘（MathPaperMiner）
**原则**：数学驱动引擎必须与时俱进，不能仅依赖初始设定的几个公式。

- **并行搜索**：系统会定期或根据指令，并行搜索全网关于 PCG、SDF、可微渲染、色彩科学的最新论文。
- **价值评估**：根据适用性、可实现性、新颖性进行打分。高分论文将被记录到 `knowledge/math_papers.md`，并作为"候选模型"加入注册表。
- **能力缺口暴露**：如果某个高分模型需要 GPU 或 Unity 环境，系统会通过 `mathart-evolve status` 明确暴露该缺口，请求你配合升级硬件或软件环境。

## 3. 扩展性预留：Unity Shader 与伪 3D

为了确保项目在后期的生命力，系统已在底层架构中预留了向高级表现形式进化的骨架。

### 3.1 Unity Shader 知识模块
- **当前状态**：系统已内置 `ShaderKnowledgeBase`，包含 URP 2D Lit、Outline、Palette Swap 等着色器的数学参数边界。
- **代码生成**：`ShaderCodeGenerator` 可以直接将蒸馏出的数学规则转换为 Unity HLSL 代码片段和 Shader Graph JSON。
- **进化路径**：当你准备好 Unity 环境后，我们可以闭环：系统生成 Shader 参数 → 你在 Unity 中运行并截图 → 系统评估截图质量 → 系统优化 Shader 参数。

### 3.2 伪 3D 表现（Pseudo-3D）
- **当前状态**：`Pseudo3DExtension` 已实现了纯数学层面的等距投影（Isometric Projection）、画家算法深度排序、2D 转法线贴图（Normal Map）近似计算以及视差滚动（Parallax）的矩阵运算。
- **进化路径**：当项目需要从纯 2D 进化到 2.5D/伪 3D 时，这些数学基础将直接驱动 Unity 的深度着色器，实现带有动态光影和透视变化的像素美术资产。

## 4. 沟通与反馈规范

1. **上传资料时**：只需上传附件并说明"蒸馏进项目"或"参考这个优化模型"，系统会自动走去重和整合流程。
2. **遇到瓶颈时**：如果我回复了 `STAGNATION_LOG.md` 的内容，请阅读其中的"诊断原因"，并告诉我你的决定（例如："放宽骨骼长度约束" 或 "这个效果可以接受，停止迭代"）。
3. **环境升级时**：当我报告"需要 GPU 才能启用可微渲染"时，如果你准备好了环境，请告诉我"本地 GPU 已就绪，推送测试代码"，我将把需要算力的部分打包推送到 GitHub 供你拉取运行。

本协议自 v0.3.0 版本起生效，并将随着项目的进化而持续更新。
